# 商品库存编辑重构临时计划

## 核心目标

库存编辑不需要一个专门处理库存的特殊界面。它应该复用和上架出售商品时相同的通用物品栏式选择界面，只是在库存编辑模式下，商品库存对应一个服务端权威的
1 格槽位。

最终规则很简单：库存编辑结束后，挂单库存数量等于这个槽位里的物品数量。

## 需求

- 玩家点击“编辑商品”按钮后，服务端立刻锁定该挂单。
- 锁定期间，其他玩家无法购买、确认、管理或访问该挂单。
- 锁在商品编辑界面关闭、返回市场主页、玩家掉线或菜单被移除时释放。
- 玩家点击“编辑库存”时，打开和上架出售商品选择物品时相同的通用物品栏式界面。
- 库存编辑界面不只是“长得像”上架选择界面，而是同一个通用界面在不同模式下工作。
- 库存编辑槽是服务端权威的普通容器槽位。
- 库存编辑槽初始数量等于当前挂单库存。
- 库存编辑槽最大数量为 `999`。
- 库存编辑槽只能放入与挂单原商品 `ItemStack.isSameItemSameComponents(...)` 匹配的物品。
- 玩家背包槽保持原版行为和原版堆叠上限。
- 玩家操作库存编辑槽时，应尽量等同普通物品栏槽位：左键、右键、Shift 移动、放入、取出都走正常容器交互。
- 库存槽交互不应在每次点击时访问数据库，也不应依赖客户端模拟库存变化。
- “确定”和“返回”按钮同义，都表示结束库存编辑并以槽内数量作为最终库存。
- ESC、关闭库存编辑界面、返回商品编辑页也使用同一语义。
- 库存编辑结束后，挂单库存数量直接等于库存编辑槽内数量。
- 如果槽内为空，则最终挂单库存为 `0`。

## 非目标

- 不新增一个只服务于库存编辑的特殊 UI 树。
- 不把库存编辑做成加减按钮或输入数字的特殊面板。
- 不让客户端提交最终库存数量作为权威值。
- 不在每次拿取库存物品时访问数据库。
- 不让被锁定的挂单继续被其他玩家购买或确认。
- 不把库存编辑实现为“客户端解释某个选择结果”的流程。

## 推荐结构

### 1. 商品编辑锁

进入商品编辑页前，客户端应先向服务端请求编辑该挂单。服务端确认玩家有权限后锁定挂单，再允许打开商品编辑页。

锁定状态最少需要记录：

- `marketItemId`
- `editingPlayerId`
- 当前菜单或编辑会话引用

所有市场访问路径都必须检查锁：

- 其他玩家购买该挂单时失败。
- 其他玩家确认该挂单订单时失败。
- 其他玩家管理该挂单时失败。
- 市场列表可以隐藏被锁挂单；即使列表没有隐藏，服务端操作也必须拒绝。

锁释放时机：

- 商品编辑界面关闭。
- 玩家返回市场主页。
- 玩家掉线。
- 菜单被移除。
- 挂单被下架。

### 2. 通用物品栏式界面

上架出售商品选择物品、收购单取样、编辑挂单库存，都应使用同一个通用物品栏式界面。

这个通用界面通过模式配置改变行为：

- 上架出售：选择玩家背包里的真实物品。
- 收购取样：选择物品样品。
- 编辑库存：显示并操作挂单库存槽。

库存编辑模式只改变标题、提示文案、槽位规则和完成动作，不改变界面类型。

### 3. 库存编辑槽

库存编辑模式下，通用界面里的主槽位代表挂单库存。

这个槽位应是服务端菜单中的真实 `Container/Slot`，而不是客户端 UI 伪造出来的显示状态。

槽位规则：

- 初始内容为挂单商品模板，数量为当前挂单库存。
- 最大数量为 `999`。
- `mayPlace(stack)` 只允许同物品、同组件。
- `mayPickup(player)` 正常允许。
- `getMaxStackSize()` 返回 `999`。
- `getMaxStackSize(stack)` 返回 `999`。
- Shift 移动在玩家背包和库存槽之间按普通容器逻辑转移。
- 玩家背包槽不改变自身最大堆叠限制。

由于商品编辑页已经锁定挂单，库存槽里的物品就是当前挂单库存的权威表现。

### 4. 库存写回语义

库存编辑不需要区分“确认”和“取消”。只要离开库存编辑模式，就读取服务端槽位最终状态。

结束库存编辑时：

- 服务端读取库存编辑槽。
- 如果槽为空，`finalStock = 0`。
- 如果槽不为空，校验槽内物品仍与挂单商品同物品同组件。
- `finalStock = 槽内物品数量`。
- 将挂单库存写为 `finalStock`。
- 清理库存编辑槽状态。
- 返回商品编辑页或市场主页。

客户端可以发送“结束库存编辑”的意图，但不能发送最终库存数量作为权威值。

### 5. 异常关闭

如果玩家在库存编辑过程中关闭界面、返回、掉线或菜单被移除，服务端也应按同一规则处理：读取槽内最终数量，写回挂单库存，然后释放挂单锁。

不能只释放锁而丢弃槽内物品，也不能让挂单保持永久锁定。

## 实现步骤

1. 为挂单新增商品编辑锁，进入商品编辑页前由服务端加锁。
2. 让购买、确认订单、管理操作、市场列表等路径统一尊重商品编辑锁。
3. 将库存编辑入口接入和上架出售相同的通用物品栏式界面。
4. 为通用物品栏界面增加库存编辑模式。
5. 在库存编辑模式下使用服务端权威的 1 格库存槽，最大数量 `999`。
6. 让“确定”“返回”、ESC、关闭都调用同一个结束库存编辑流程。
7. 结束库存编辑时，服务端将挂单库存写为槽内数量。
8. 商品编辑页关闭、玩家掉线或菜单移除时，释放挂单锁。

## 验证场景

- 玩家进入自己的商品编辑页后，其他玩家无法购买该挂单。
- 玩家进入自己的商品编辑页后，其他玩家无法确认该挂单订单。
- 玩家进入自己的商品编辑页后，其他玩家无法管理该挂单。
- 商品编辑页关闭后，挂单锁被释放。
- 玩家掉线后，挂单锁被释放。
- 点击库存编辑后，打开的是和上架出售选择物品同一个通用界面。
- 库存编辑槽初始数量等于挂单库存。
- 左键取出、右键取出、放回、Shift 移动行为接近普通容器槽。
- 库存槽达到 `999` 后继续放入失败且不吞物。
- 玩家背包槽仍然遵守原版堆叠限制。
- 放入不同组件的物品失败且不吞物。
- 点击“确定”后，挂单库存等于槽内数量。
- 点击“返回”后，挂单库存同样等于槽内数量。
- ESC 或关闭界面后，挂单库存同样等于槽内数量。
- 槽为空时，挂单库存变为 `0`。

## 临时结论

这个方案的核心不是专用库存编辑界面，而是服务端锁和普通容器槽。

只要玩家进入商品编辑页时挂单已经被服务端锁定，库存编辑就可以安全复用通用物品栏式界面。库存数量不需要额外解释，也不需要客户端提交权威数值：库存槽里有多少，挂单库存就是多少。
