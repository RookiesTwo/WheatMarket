# 商品库存编辑重构临时计划

## 背景

当前“物品选择”界面同时承担上架物品选择、收购取样、商品库存编辑等职责。库存编辑场景中，市场库存被伪装成选择槽里的 `ItemStack`
，再通过额外状态区分玩家真实放入的物品、市场虚拟库存、已经临时取出的库存。这让菜单同步、客户端预测、关闭回滚、确认提交都变复杂，也容易出现取出延迟、回弹、吞物或重复发放风险。

目标是把“商品库存编辑”从通用物品选择流程中拆出来，改成一个服务端权威的专用临时库存容器。

## 目标行为

- 编辑出售单库存时，打开一个专用库存编辑界面。
- 界面里有一个商品库存槽，槽内初始数量等于当前出售单剩余库存。
- 该槽像普通箱子槽一样交互：左键、右键、Shift 移动、放入、取出都应符合 Minecraft 容器行为。
- 该槽最大堆叠数量为 `999`。
- 玩家背包槽仍保持原版限制，不突破 64 上限。
- 槽内只能放入与原商品 `ItemStack.isSameItemSameComponents(...)` 匹配的物品。
- 结束编辑时，不区分“确认”和“返回”的语义；只根据槽内最终数量调整商品库存。
- 槽内最终数量大于打开时数量：表示补货，多出来的物品来自玩家背包。
- 槽内最终数量小于打开时数量：表示取出，少掉的物品已经进入玩家背包。
- 槽内最终数量等于打开时数量：不提交库存变化。

## 非目标

- 不继续把库存编辑塞进通用 `ItemSelectionRequest` / `ItemSelectionResult` 流程。
- 不依赖客户端本地预填市场库存。
- 不把数据库库存伪装成玩家已拥有物品再通过补偿状态回收。
- 不在库存编辑槽里支持收购单取样逻辑。

## 推荐架构

### 1. 新增库存编辑会话状态

在服务端菜单或独立会话对象中保存：

- `marketItemId`
- `originalStockAmount`
- `stockEditContainer`
- `lockedStackTemplate`
- `dirty/finalized` 标记

`stockEditContainer` 是一个 1 格 `Container`，初始内容为原商品模板，数量为 `originalStockAmount`，最大数量 `999`。

### 2. 新增专用槽位类型

新增类似 `StockEditSlot extends Slot` 的槽：

- `mayPlace(stack)` 只允许同物品、同组件。
- `mayPickup(player)` 正常允许。
- `getMaxStackSize()` 返回 `999`。
- `getMaxStackSize(stack)` 返回 `999`。

玩家背包槽继续用普通玩家背包 slot，不改变上限。

### 3. 新增或复用专用 Screen/UI

推荐新增库存编辑子界面，而不是复用通用物品选择 UI：

- 展示商品信息、当前编辑槽、玩家背包。
- 提供“完成/返回”按钮即可。
- 关闭、ESC、返回按钮都走同一套 finalize 逻辑。
- UI 文案强调“槽内数量就是最终库存”。

### 4. 新增 C2S finalize 包

新增类似 `FinalizeStockEditC2SPacket`：

- 客户端只发送“结束编辑”的意图。
- 服务端从当前菜单的 `stockEditContainer` 读取最终数量。
- 服务端自己计算 `delta = finalAmount - originalStockAmount`。
- 服务端校验当前玩家是否仍有权限编辑该商品。
- 服务端校验商品仍存在、仍是出售单、未变成无限库存。
- 服务端校验槽内物品模板匹配原商品模板。

不要让客户端发送最终数量作为权威值。

## 数据一致性策略

### 推荐策略：编辑会话持有库存

打开编辑时，服务端将商品当前库存转入编辑会话：

- 数据库商品库存可以临时锁定，或标记为编辑中。
- 会话槽成为这批库存的权威容器。
- 完成编辑时，将槽内剩余数量写回数据库。
- 取消/断线/异常关闭时，也将槽内剩余数量写回数据库。

优点：

- 不需要维护“虚拟已取出但尚未提交”的补偿状态。
- 玩家拿到物品后就是正常容器交互结果。
- 关闭和提交都只看槽内最终数量。

风险：

- 需要防止其他玩家在编辑期间购买同一出售单。
- 需要处理编辑会话超时、玩家断线、服务器关闭。

### 备选策略：不锁库存，只在 finalize 时提交 delta

打开时只把当前库存复制到编辑槽，完成时提交 delta。

不推荐，因为玩家已从槽里拿到物品后，如果数据库提交失败，回滚会很复杂，容易出现物品和数据库不一致。

## 主要实现步骤

1. 保留当前通用物品选择界面给上架出售和收购取样使用。
2. 从 `EDIT_LISTING_STOCK` 路径中移除对 `WheatMarketItemSelectionScreen` 的复用。
3. 新增库存编辑专用会话/容器/槽位。
4. 新增库存编辑 UI 或 Screen，用专用槽展示市场库存。
5. 新增打开库存编辑的 C2S/S2C 或复用当前已持有的商品摘要打开方式，但最终状态必须由服务端菜单管理。
6. 新增 finalize C2S 包，服务端根据槽内最终状态写回数据库。
7. 删除 `WheatMarketMenu` 中为库存编辑引入的虚拟库存临时记账逻辑。
8. 保留 `WheatMarketMenu` 中防止通用物品选择槽吞物的非标准点击保护。
9. 更新商品编辑页文案：目标库存改为“槽内数量为最终库存”或类似表述。
10. 补充断线/关闭处理，确保会话库存不会丢失。

## 需要重点验证的场景

- 打开库存编辑后直接左键取出 1 组物品。
- 右键取半组或单个物品。
- Shift 从库存槽移动到玩家背包。
- Shift 从玩家背包补货到库存槽。
- 背包满时取出库存。
- 槽内数量超过 64 但不超过 999 时渲染和交互正常。
- 槽内达到 999 后继续放入同类物品失败且不吞物。
- 放入不同 NBT/components 的同物品失败且不吞物。
- 关闭界面、按 ESC、点击返回都能写回一致库存。
- 玩家断线后库存不会消失。
- 库存编辑期间其他玩家购买同商品时被阻止或正确处理。
- 数据库操作失败时玩家物品和市场库存不会产生重复或丢失。

## 迁移注意事项

- 当前 `ManageItemC2SPacket.ACTION_RESTOCK` / `ACTION_WITHDRAW` 可以保留给旧入口或后续管理操作，但库存编辑专用流程最好使用新的
  finalize 包。
- 如果短期不做商品锁，至少要在 finalize 时重新读取数据库库存，并在库存已变化时拒绝提交，同时把编辑槽内容安全归还/写回。
- 通用物品选择界面仍可保留 `TRANSFER` 和 `SAMPLE` 两种模式，但不要再承担“市场库存容器”职责。

## 临时结论

这个方案可行，并且比继续在通用物品选择槽里维护虚拟库存状态更稳定。推荐下一步先实现服务端权威的 `StockEditContainer` 和
`StockEditSlot`，再把商品编辑页的“编辑库存”入口切换到专用库存编辑界面。
