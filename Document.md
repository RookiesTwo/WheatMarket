# WheatMarket 需求文档

## 1. 项目概述

**WheatMarket** 是一款 Minecraft 市场 MOD，玩家可在游戏内通过“笔记本电脑”方块访问虚拟市场，进行物品买卖、货币交易等操作。

当前项目为 NeoForge 单平台单模块项目，目标 MC 1.21.1，Java 21。主要依赖 LDLib2、H2 嵌入式数据库。经济与市场数据通过 H2
持久化，专用服务器路径位于 `./config/WheatMarket/database/WheatMarketDB`，单人/客户端路径位于对应存档目录下。

## 3. 待实现功能需求

### 3.1 界面需求

#### 3.1.1 主页面完善

**当前状态**：LDLib2 市场主界面已显示真实市场列表，并已接入市场列表网络请求。筛选、搜索、排序、分页、只看自己商品和余额显示均已有实现。

**剩余需求**：

- 商品卡片补充展示库存数量、卖家/系统商店标识、上架时间、最后交易时间、冷却/限购状态等关键信息
- 系统商店商品增加更明确的视觉标识，例如边框、角标或图标
- 上架按钮已进入完整上架流程，并会通过通用物品选择子界面回填物品草稿；后续仍需继续配合收购模式完整业务和跨界面状态保留做体验收尾
- 商品卡片点击后的行为区分出售单和收购单：出售单进入购买确认，收购单进入供货/成交流程
- 主界面接收到成功操作结果后会刷新当前市场列表；未来详情/管理界面仍需做局部刷新或返回后刷新
- 卖家名称解析需要更完整，避免只能显示 UUID 片段

#### 3.1.2 出售单购买确认界面

**当前状态**：已实现出售单购买确认界面。界面显示商品、卖家/系统、单价、名称、上架时间、库存、余额、冷却/限购提示；支持输入数量、加减按钮、确认购买、成功签名/盖章动画；确认后发送
`BuyItemC2SPacket`。普通玩家打开自己发布的出售单时，主按钮会改为“编辑商品”；OP 打开其他商品时会额外显示“管理”入口，并可进入商品编辑界面。

**剩余需求**：

- 根据余额在客户端提前禁用或提示不可购买，当前主要依赖服务端失败结果
- 背包满时目前会把物品掉落到地上，仍需按需求改为明确提示或可配置处理
- 冷却失败提示可进一步显示剩余可购买数量和剩余时间
- 支持从确认界面返回时保留主页面筛选/分页状态
- 收购单在当前确认界面仍只提示暂不支持成交，后续需接入供货/成交流程

#### 3.1.3 上架界面

**当前状态**：`ListItemC2SPacket` 和 `MarketService.listItem(...)` 已接入完整上架流程。主页面“上架”按钮现在会进入
`WheatMarketListingScreen` / `WheatMarketListingUI` 组成的完整表单，支持价格、数量、交易类型和 OP 选项；通用 LDLib2
物品选择子界面作为物品字段编辑器，通过 `ItemSelectionRequest` / `ItemSelectionResult` 回填物品模板和数量，并在确认后发送
`ListItemC2SPacket`。出售单上架成功后会扣减实际物品；收购单可创建基础记录，但后续业务尚未完整实现。

**剩余需求**：

- 收购模式仍需补完资金预留、供货物品校验、成交后货币流向、待领取托管和退款/过期规则，不能沿用出售模式的扣物品逻辑
- 收购单创建请求仍不能信任客户端传来的任意 `itemNBTCompound`；服务端应继续以玩家背包/菜单中的真实样品为准生成
  `itemNBTCompound`
- 上架前后仍需持续校验物品槽位、数量、NBT/components 和物品一致性，尤其是收购模式与系统商品路径

#### 3.1.4 商品详情及商品管理界面

**当前状态**：`OrderConfirmationScreen` 已同时承担商品详情和商品编辑入口。普通玩家打开自己发布的出售单时，主按钮会变为“编辑商品”；OP
打开其他商品时会额外显示“管理”入口。`WheatMarketItemEditScreen` / `WheatMarketItemEditUI` 已接入改价、库存编辑、下架和 OP
专属选项；库存编辑通过编辑锁和专门的库存编辑子流程实现，物品选择界面会预填当前库存并锁定原商品模板。

**剩余需求**：

- 收购单管理当前仍未完成；确认界面和编辑界面都还不支持收购单成交、库存编辑和下架的完整流程
- 取出和下架时仍需处理背包满的情况，避免继续直接掉落或给出不明确反馈
- 支持从详情/管理界面返回时保留主页面筛选/分页状态
- 如后续需要更强的实时一致性，可继续补充详情界面局部刷新，或新增独立商品详情 request/response 包

### 3.2 数据与网络需求

#### 3.2.1 网络通信系统

**当前状态**：已迁移到 NeoForge Custom Payload API。当前注册的包包括：

- C2S：`RequestMarketListC2SPacket`、`BuyItemC2SPacket`、`ListItemC2SPacket`、`ManageItemC2SPacket`、
  `AcquireItemEditLockC2SPacket`、`ReleaseItemEditLockC2SPacket`、`SetItemSelectionModeC2SPacket`、
  `BeginStockEditC2SPacket`、`FinalizeStockEditC2SPacket`
- S2C：`MarketListS2CPacket`、`OperationResultS2CPacket`、`BalanceUpdateS2CPacket`

**剩余需求**：

- 上架界面已接入 `ListItemC2SPacket`，商品详情/管理界面已接入 `ManageItemC2SPacket`；库存编辑子流程当前由
  `BeginStockEditC2SPacket` / `FinalizeStockEditC2SPacket` 负责开始和提交
- `SetItemSelectionModeC2SPacket` 继续负责同步物品选择菜单的固定模式、初始选择和锁定模板；最终上架/库存编辑提交仍必须由对应业务
  packet 和服务端校验完成
- 当前 `MarketListS2CPacket` 已补足详情界面所需核心字段；如未来详情需要更强实时数据，再考虑新增独立商品详情
  request/response 包
- 继续完善跨界面的主动刷新、局部刷新和状态保留，避免只依赖重建页面或玩家手动刷新
- 收购单成交需要新增或扩展网络包，避免与出售购买流程混用
- 收购单创建需要让客户端传递样品来源槽位或通过真实菜单槽位同步，服务端必须以玩家背包中的实际样品为准生成
  `itemNBTCompound`

#### 3.2.2 数据库 CRUD 和查询能力

**当前状态**：`market_item` repository 只有基础 CRUD；市场列表的条件过滤、搜索、排序和分页当前在 `MarketService` 基于
`MarketItemCache` 完成。`purchase_record` 已支持当前冷却系统需要的时间窗口查询。

**剩余需求**：

- 明确是否继续以缓存为主做市场查询；如果市场规模扩大，需要补充 repository 层条件查询和分页 SQL
- `MarketItemRepository` 可补充按条件查询、分页查询、可用商品查询、过期商品查询
- `PurchaseRecordRepository` 补充按买家/商品查询交易记录列表，用于历史记录和管理界面
- 保持缓存只在 service 事务成功后更新，不恢复“缓存全量落库”作为主要持久化机制
- 库存为 0 的商品应自动下架或至少从市场列表隐藏

#### 3.2.3 收购单待领取物品托管

**当前状态**：未实现。收购单发布者离线时，其他玩家向该收购单出售物品后，不能直接把物品写入离线玩家背包。

**设计要求**：

- 收购单成交所得物品进入市场托管仓库，由收购单发布者上线后在自建领取界面中领取
- 托管仓库以 H2 数据库为权威存储；界面只作为托管记录的视图，不使用玩家 `Inventory`、临时 `SimpleContainer` 或方块背包作为持久化来源
- 领取时由 service 执行校验和事务：确认托管记录存在、玩家背包可容纳、发放物品、删除或减少托管记录
- 背包满时不删除托管记录；可以拒绝领取，或只领取可放入数量并保留剩余数量
- 收购单取消或过期时，只退还未成交部分的冻结资金；已成交但未领取的物品继续保留在托管仓库

**建议表结构**：

- `deliveryID VARCHAR(36) PRIMARY KEY`
- `receiverID VARCHAR(36) NOT NULL`：待领取玩家，即收购单发布者
- `marketItemID VARCHAR(36)`：来源收购单，可为空以兼容未来迁移或被删除的订单
- `sourcePlayerID VARCHAR(36)`：供货玩家
- `itemID VARCHAR(255) NOT NULL`：单独保存物品 ID，便于查询、展示、统计和迁移
- `itemNBTCompound CLOB`：沿用现有项目字段名，保存 `ItemStack.save(...)` 得到的完整物品栈数据
- `amount INT NOT NULL`：单独保存数量，便于部分领取和避免只为数量解析 NBT
- `createdTime TIMESTAMP DEFAULT CURRENT_TIMESTAMP`
- `claimedTime TIMESTAMP` 或 `remainingAmount INT`：根据实现选择“整条领取删除”或“允许部分领取”

#### 3.2.4 数据库操作异步化与队列

**当前状态**：经济和市场写操作已通过 `CompletableFuture` + 单线程 DB executor 串行执行；服务器关闭时会等待 executor
结束并关闭数据库连接。

**剩余需求**：

- 避免高频市场列表请求在主线程执行过重同步查询
- 服务器关闭前需要更明确地保证未完成 DB 操作完成或可追踪失败
- 为关键异步操作补充日志和失败回调，便于排查数据不一致

#### 3.2.5 数据库存档保存

**当前状态**：经济和市场数据在业务发生时通过 service/repository 事务提交；世界保存不是主要持久化机制。

**剩余需求**：

- 可在世界保存或定时任务中执行过期商品清理、统计数据写入、缓存/数据库一致性检查
- 服务器关闭前应尽量完成所有未完成数据库操作

### 3.4 经济系统需求

#### 3.4.1 货币系统

**当前状态**：`player_info` 表存储余额，`/balance`、`/pay`、`/account` 已实现。出售单购买时会扣减买家余额；非系统商店会增加卖家余额，系统商店当前相当于销毁买家支付的货币。

**剩余需求**：

- 初始余额可配置，当前固定为 0
- 系统商店货币流向可配置：销毁、注入指定账户、进入公共账户等
- 收购模式需要定义资金预留和成交后的余额变化
- 继续保持 service 层拒绝 `NaN`、`Infinity`、负数和其他非法金额

#### 3.4.2 冷却和限购系统

**当前状态**：出售单购买时已根据 `cooldownAmount` 和 `cooldownTimeInMinutes` 查询 `purchase_record`
，并阻止超出限制的购买。市场列表也会返回当前玩家在时间窗口内的剩余可购买数量。

**剩余需求**：

- 改善客户端展示，明确显示限购数量、周期、剩余额度和失败原因
- 支持收购单成交的冷却/限购判断
- 补充更完整的测试覆盖，尤其是时间窗口边界和多次购买累计数量

#### 3.4.3 商品过期

**当前状态**：`MarketItem.isExpired()` 已实现；市场列表会过滤过期商品，购买时也会拒绝过期商品。

**剩余需求**：

- 过期商品自动下架
- 玩家商品过期后库存退回卖家，或设计离线退回/邮件/临时仓库机制
- 系统商店过期后的处理规则需要单独定义

### 3.5 收购模式需求

**当前状态**：数据模型中 `ifSell` 字段可区分出售/收购；主界面可以筛选和显示收购标签；购买确认界面遇到收购单会提示暂不支持成交。当前上架
packet 虽可传 `ifSell=false`，但收购单的经济、物品流向、资金冻结和待领取托管尚未完整实现。

**需求**：

- 出售模式：卖家挂出物品，买家支付货币获得物品
- 收购模式：买家挂出收购单，供货玩家提供物品并获得货币
- 收购单上架时必须从玩家物品栏样品框读取精确样品，创建目标物品模板
- 收购单样品不扣除，只用于保存 `itemID` 和 `itemNBTCompound`；`amount` 表示剩余收购需求量
- 收购单只支持精确匹配，供货时必须校验提交物品与收购单样品模板一致
- 收购单上架时需要冻结或校验发布者资金
- 成交后从预留资金或发布者账户向供货玩家付款
- 成交后物品写入待领取托管仓库，归属给收购单发布者，不能直接写入离线玩家背包
- 收购单发布者通过自建领取界面领取托管物品；领取失败时托管记录必须保留
- 收购单的库存/需求量、冷却、过期、退款规则需要与出售单分开定义

## 5. 已知问题

1. 上架完整表单和商品详情/管理界面已接入，但收购模式的上架后业务、成交流程和托管仍未完整实现。
2. 出售单购买确认界面已实现，但客户端余额预校验、失败提示细化、返回状态保留仍未完成；收购单成交仍未实现。
3. 收购单待领取物品托管仓库和领取界面仍未实现，离线收购成交规则尚未落地。
4. 商品卡片目前信息较少，未直接展示库存、卖家、上架时间、系统商店标识等完整字段。
5. 有限库存商品售罄后已自动删除挂单；过期商品自动下架和库存退回仍未实现。
6. “无限数量”已改为独立 `ifInfinite` 字段；旧 `Integer.MAX_VALUE` 数据仅作为启动迁移兼容来源。
7. 背包满时购买、取出或下架当前可能掉落物品，尚未按需求提供明确提示或可配置处理。
8. 货币继续使用 `double` / SQL `DOUBLE`，必须保持 service 层拒绝 `NaN`、`Infinity` 和非法金额。
9. 物品数据仍以 `ItemStack.save(...)` 得到的 NBT 写入数据库，缺少统一稳定的 `ItemStack` codec/迁移策略。
10. `market_item` repository 仍未提供完整条件查询/分页查询，当前依赖 service/cache 层过滤。
11. Config 商品列表、过期商品退回、服务器商店货币流向配置仍未实现。
