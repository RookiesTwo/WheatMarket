# WheatMarket 需求文档

## 1. 项目概述

**WheatMarket** 是一款 Minecraft 市场MOD，玩家可在游戏内通过"笔记本电脑"方块访问一个虚拟市场，进行物品买卖、货币交易等操作。MOD
当前为 NeoForge 单平台单模块项目（MC 1.21.1，Java 21），依赖 LDLib2，内置 H2 嵌入式数据库存储经济数据。

## 3. 待实现功能需求

### 3.1 界面需求

#### 3.1.1 主页面（已有框架，需完善）

**当前状态**：标题栏、筛选区、搜索栏已搭建，商品区域使用占位纸片渲染。

**需求**：
- 商品区域需替换占位符，展示真实的 `MarketItem` 列表
- 每个商品条目应显示：物品图标、物品名称、单价、数量、卖家名称、上架时间
- 系统商店商品应有视觉标识（如特殊边框或图标）
- 支持翻页/滚动浏览
- 筛选逻辑需实际生效（按交易类型、物品来源、排序方式过滤）
- 搜索按钮需实现按物品名称/ID搜索

#### 3.1.2 商品子界面及商品设置界面（核心需求）

**来源**：todoList 第2条

**需求**：
- 点击主页面商品条目后进入此界面
- 显示该商品的详细信息：物品图标、名称、描述、价格、数量、卖家、上架时间、最后交易时间
- **显示玩家物品栏**，以便进行补货和取出操作
- 操作功能：
  - **补货**：从玩家物品栏向该商品添加库存
  - **取出存货**：从商品库存取出物品到玩家物品栏
  - **编辑价格**：修改商品的出售/收购价格
  - **下架**：将该商品从市场中移除，剩余库存退回玩家
- **OP 专属操作**：
  - **设置是否为服务器商店**：切换 `ifAdmin` 标志
  - **设置是否为无限数量**：启用后商品数量不减少
  - **设置冷却时间**：配置 `cooldownAmount` 和 `cooldownTimeInMinutes`
- 权限控制：仅卖家本人或 OP 可操作此界面

#### 3.1.3 购买商品界面（核心需求）

**来源**：todoList 第3条

**需求**：
- 从主页面点击"购买"后进入此界面
- 显示商品信息摘要
- **数量选择**：支持输入或 +/- 按钮调整购买数量，上限为商品可用数量
- 显示总价格 = 单价 × 数量
- 显示玩家当前余额，判断是否可购买
- **确认按钮**：扣除余额、扣减库存、生成购买记录
- 购买后物品进入玩家物品栏（背包满时提示）
- 冷却限制检查：若商品有冷却，需判断玩家是否在冷却期内

#### 3.1.4 上架界面（核心需求）

**来源**：todoList 第4条

**需求**：
- 从主页面点击"上架"后进入此界面
- **显示玩家物品栏**，玩家可选择要上架的物品
- **价格编辑**：输入出售/收购价格
- 物品选择：从物品栏中选择物品和数量
- **OP 专属选项**：
  - 选择是否为服务器商店
  - 设置是否为无限数量
  - 设置冷却时间
- 上架成功后创建 `MarketItem` 记录并存入数据库
- 从玩家物品栏扣减对应物品（服务器商店除外）

### 3.2 数据与网络需求

#### 3.2.1 网络通信系统

**当前状态**：已迁移到 NeoForge Custom Payload API，注册了市场列表请求、购买、上架、商品管理、市场列表返回、操作结果、余额更新等
C2S/S2C 包。但当前 GUI 尚未调用 `WheatMarketNetwork.sendToServer(...)`，市场界面仍未真正触发服务端交互。

**需求**：

- 将 GUI 操作接入现有 NeoForge payload 网络
- **C2S 包（客户端→服务端）**：
  - 请求市场列表（含筛选/搜索/分页参数）
  - 请求购买商品（商品ID、数量）
  - 请求上架商品（物品NBT、价格、数量、选项）
  - 请求商品管理操作（补货、取出、改价、下架）
- **S2C 包（服务端→客户端）**：
  - 市场列表数据（商品摘要列表）
  - 商品详情数据
  - 操作结果反馈（成功/失败消息）
  - 余额更新通知

#### 3.2.2 数据库 CRUD 补全

**当前状态**：`market_item` 已通过 repository/service 覆盖现有上架、购买、管理和基础列表路径；`purchase_record` 已有 insert
和按买家/商品查询最近购买时间。仍缺少条件查询、分页查询、按买家/商品查询记录列表等完整查询能力。

**需求**：

- `MarketItemRepository`：补全按条件查询、分页查询、过期/可用商品查询
- `PurchaseRecordRepository`：补全按买家/商品查询记录列表
- 继续保持缓存只在 service 事务成功后更新，不恢复缓存全量落库作为主要持久化机制

#### 3.2.3 数据库操作异步化与队列

**来源**：todoList 第7条

**需求**：
- 所有数据库写操作通过异步队列处理，避免阻塞主线程
- 使用 `CompletableFuture` + 单线程 DB executor，实现串行任务队列
- 写操作入队后立即返回，服务端逻辑通过回调或事件获取结果
- 保证写操作的顺序性和一致性

#### 3.2.4 数据库存档保存

**当前状态**：经济和市场写操作已在业务发生时通过 service/repository 事务提交；服务器关闭时关闭数据库连接和 DB executor。

**需求**：
- 世界保存不作为经济数据的主要持久化机制
- 可在世界保存时补充执行过期商品清理、统计数据写入或缓存/数据库一致性校验
- 确保服务器关闭前完成所有未完成的数据库操作

### 3.3 整合包支持需求

#### 3.3.1 Config 配置商品列表

**来源**：todoList 第6条

**需求**：
- 支持从 config 文件读取预定义的商品列表
- 格式建议使用 JSON 或 TOML，每个条目包含：物品ID、NBT（可选）、价格、数量、是否服务器商店、是否无限数量、冷却设置
- 服务器启动时自动加载配置，将配置中的商品注册到市场（若不存在）
- 支持热重载配置（通过命令）

### 3.4 经济系统需求

#### 3.4.1 货币系统

**当前状态**：`player_info` 表存储余额，`/balance`、`/pay`、`/account` 命令已实现。

**需求**：
- 初始余额可配置（当前代码中默认为 0）
- 购买时自动扣减买家余额、增加卖家余额
- 服务器商店的货币流向可配置（销毁/注入系统）

#### 3.4.2 冷却系统

**需求**：
- 根据商品的 `cooldownAmount` 和 `cooldownTimeInMinutes` 判断玩家是否可购买
- 查询 `purchase_record` 表判断玩家上次购买时间
- 冷却期内阻止购买并提示剩余冷却时间

#### 3.4.3 商品过期

**需求**：
- `MarketItem.isExpired()` 已实现，需在查询时过滤过期商品
- 过期商品自动下架，库存退回卖家

### 3.5 收购模式需求

**当前状态**：数据模型中 `ifSell` 字段区分出售/收购，但界面和逻辑未实现。

**需求**：
- 出售模式（ifSell=true）：卖家挂出物品，买家支付货币获得物品
- 收购模式（ifSell=false）：买家挂出收购单，卖家提供物品获得货币
- 收购界面的交互逻辑与出售相反：提供物品 → 获得货币

### 3.6 其他需求

#### 3.6.1 交易板方块

**当前状态**：模型和贴图已存在，但无 Java 类和注册。

**需求**：
- 实现 `TradingBoardBlock`，可作为墙壁挂载方块
- 右键打开市场界面（与 Laptop 相同功能）
- 或设计为仅显示市场公告/热门商品的只读界面

#### 3.6.2 缺失的语言键补全

**需求**：
- 补全 `gui.wheatmarket.searchbar` 等缺失键
- 为所有新增界面和功能添加中英文翻译
